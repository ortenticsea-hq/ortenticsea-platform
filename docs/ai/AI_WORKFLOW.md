# Ortenticsea – AI Workflow

This document defines **how AI is used operationally** within the Ortenticsea codebase.

It establishes a clear, repeatable workflow that ensures:

* High code quality
* Predictable agent behavior
* Safe collaboration between humans and AI
* Long-term maintainability of the product

This document works in conjunction with:

* `/docs/ai/SYSTEM_PROMPT.md`
* `/docs/ai/AGENT_RULES.md`

---

## Purpose

AI is treated as a **force-multiplier**, not an autonomous decision-maker.

All AI contributions must:

* Be intentional
* Be reviewable
* Respect architecture and ownership boundaries

Ortenticsea prioritizes **clarity, safety, and maintainability** over speed.

---

## Approved AI Use Cases

AI may be used to:

* Scaffold features and modules
* Generate boilerplate code
* Implement clearly defined requirements
* Refactor code within a single responsibility scope
* Generate or update documentation
* Identify edge cases and testing gaps

AI must NOT be used to:

* Make product or business decisions autonomously
* Introduce new frameworks or platforms without approval
* Perform sweeping refactors without instruction
* Commit code without human review

---

## Standard AI Development Flow

### 1. Define Intent

Before invoking AI, clearly define:

* The problem being solved
* The affected scope (frontend, backend, docs)
* Constraints and non-goals

Ambiguous intent must be clarified before code generation begins.

---

### 2. Select the Appropriate Agent

Choose the agent based on scope:

* **Frontend Agent** → UI, UX, client state
* **Backend Agent** → APIs, domain logic, data
* **Documentation Agent** → README, architecture, decisions

Agents must follow `/docs/ai/AGENT_RULES.md` strictly.

---

### 3. Provide System Context

Every AI session must be primed with:

* `/docs/ai/SYSTEM_PROMPT.md`
* `/docs/ai/AGENT_RULES.md`
* Relevant architecture or feature documentation

This ensures consistent behavior across tools and environments.

---

### 4. Generate Incremental Changes

AI output should:

* Focus on a single, well-defined task
* Avoid touching unrelated files
* Produce code that is readable and testable
* Include error handling and edge cases

Large features should be broken into smaller, sequential steps.

---

### 5. Human Review (Mandatory)

All AI-generated changes must be reviewed by a human before merging.

Review checklist:

* Does this follow SYSTEM_PROMPT.md?
* Does it respect AGENT_RULES.md?
* Is the architecture preserved?
* Are edge cases handled?
* Is documentation updated if required?

AI output is considered **draft-quality** until reviewed.

---

### 6. Documentation Synchronization

When functionality changes:

* README.md must be updated if user-facing behavior changes
* `/docs/architecture.md` must be updated if structure changes
* Assumptions and trade-offs must be documented

Documentation drift is treated as a defect.

---

## Version Control & Commits

* AI-generated changes should be committed in small, logical units
* Commit messages must describe **intent**, not tool usage

  * Good: `Add product verification API`
  * Bad: `Generated by AI`
* Avoid mixing AI-generated and manual changes in the same commit when possible

---

## Conflict Resolution

If AI output conflicts with:

1. SYSTEM_PROMPT.md
2. AGENT_RULES.md
3. Architecture documentation

The AI output must be corrected or discarded.

When uncertainty exists, **pause and clarify** before proceeding.

---

## Continuous Improvement

This workflow is a living system.

Improvements should:

* Be documented in this file
* Include rationale and trade-offs
* Be reviewed like code changes

The goal is to evolve Ortenticsea’s AI usage responsibly as the product scales.

---

## Final Principle

> AI accelerates delivery, but **engineering judgment remains human-owned**.

This workflow ensures Ortenticsea benefits from AI without sacrificing quality, security, or long-term vision.
